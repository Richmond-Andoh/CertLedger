1. ENTITY RELATIONSHIP DIAGRAM (ERD) — DATABASE SCHEMA DESIGN
================================================================================

The Entity Relationship Diagram (ERD) illustrates the persistent storage layer 
of CertLedger, mapped to a Document-Oriented Model (MongoDB). It specifies how 
the physical data is structured across collections and how they relate.

1.1 Core Entities & Attribute Details
-------------------------------------
*   USER COLLECTION (System Administrators, University Admins, and Students):
    - _id (ObjectId, Primary Key): Unique internal database identifier.
    - username (String, Unique): System identity login handle.
    - email (String, Unique): Academic or administrative contact email.
    - password (String): Securely hashed credential using bcrypt.
    - role (Enum): System role defining access boundaries [SystemAdmin, UniversityAdmin, Student].
    - institution (String): Name of the university/college the user belongs to (null for SystemAdmin).
    - tempPassword (String): Plain temporary password generated for new students (cleared upon first login).
    - transactionHash (String): The transaction hash that recorded this user's node creation on-chain (if applicable).
    - isActive (Boolean): Control flag for account suspension or deactivation.
    - createdAt / updatedAt (Date): Autogenerated tracking timestamps.

*   CERTIFICATE COLLECTION (Academic Records):
    - _id (ObjectId, Primary Key): Unique database identifier.
    - certificateId (String, Unique Key): Unique institutional identifier (e.g., Degree ID).
    - hash (String, SHA-256 Unique Key): Cryptographic hash of the certificate's details, representing the core integrity proof anchored on the blockchain.
    - studentName (String): Full name of the graduate.
    - studentId (String): Unique institutional ID of the student.
    - institution (String): The issuing higher education institution.
    - qualification (String): Degree, Diploma, or Certificate earned (e.g., BSc. Computer Science).
    - issueDate (Date): Date of academic completion.
    - grade (String): Division or GPA earned (e.g., First Class Honours).
    - issuer (String, Foreign Key -> User._id): Reference to the University Admin account that authorized the upload.
    - transactionHash (String, Unique Key): Sepolia Ethereum transaction hash generated when anchoring this certificate.
    - blockNumber (Number): Ethereum ledger block in which the anchoring transaction was committed.
    - isActive (Boolean): Integrity status. Allows revoking/disabling certificates locally.

*   ANOMALY COLLECTION (Security and Integrity Auditing):
    - _id (ObjectId, Primary Key): Unique database identifier.
    - certificateId (String, Foreign Key -> Certificate.certificateId): Reference to the affected academic record.
    - type (Enum): Anomaly type [DuplicateHash, SuspiciousIssuer, NodeAddressMismatch, LocalTamper].
    - reason (String): Detailed text describing why the security anomaly was flagged.
    - severity (Enum): Level of severity [Low, Medium, Critical].
    - reporter (String): Name/ID of the actor or process that triggered the anomaly detection logic.
    - reporterAddress (String): Blockchain wallet or IP address of the agent initiating the verification.
    - resolved (Boolean): Flag representing whether the System Admin has audited and cleared the issue.
    - createdAt / updatedAt (Date): Auditing timeline tracking.

1.2 Entity Relationships & Business Logic
-----------------------------------------
*   USER to CERTIFICATE (1 : N):
    - One University Admin (User) can issue and anchor many (N) Certificates.
    - A Certificate must be linked to exactly one Issuer (User).
    - Relationship role: "issues".

*   CERTIFICATE to ANOMALY (1 : N):
    - A Certificate can trigger multiple (N) Anomaly reports if verification 
      checks fail repeatedly or flag integrity mismatches (e.g., if the local database
      record differs from the on-chain SHA-256 hash or if there is a duplicate student ID).
    - An Anomaly must point to exactly one academic Certificate.
    - Relationship role: "triggers".

2. DATA FLOW DIAGRAM (DFD) LEVEL 0 — CONTEXT DIAGRAM
================================================================================

The DFD Level 0 represents the system boundary. It visualizes CertLedger as a 
single process, showing how external entities feed data into the platform and
receive output in return.

2.1 External Entities
---------------------
1.  University Admin: Authorized institutional representatives who upload and verify student records.
2.  System Admin: IT administrators responsible for managing the platform node's overall health and authorizing university nodes.
3.  Student: The direct consumer of the credential, logging in to view and share their certificate.
4.  Public Employer: Third-party verifiers checking the authenticity of submitted academic credentials.
5.  Sepolia Blockchain (Ethereum Testnet): The decentralized ledger providing absolute integrity and immutability.

2.2 Precise Data Flow Steps
---------------------------
*   Flow 1 (University Admin -> CertLedger): Sends student certificate metadata (names, IDs, grade, division).
*   Flow 2 (CertLedger -> University Admin): Returns login credentials, system receipts, and the on-chain transaction hash.
*   Flow 3 (System Admin -> CertLedger): Inputs administrative specs (authorizing university wallets, user suspension commands).
*   Flow 4 (CertLedger -> System Admin): Returns system audit logs, active user indexes, and live security anomaly tallies.
*   Flow 5 (CertLedger -> Student): Delivers academic records, verified digital certificates, and blockchain receipts.
*   Flow 6 (Student -> CertLedger): Sends login credentials to authenticate access.
*   Flow 7 (Public Employer -> CertLedger): Submits verification fields (certificate ID or uploaded certificate files).
*   Flow 8 (CertLedger -> Public Employer): Returns an absolute, consensus-backed validation status (Genuine vs. Tampered/Fake).
*   Flow 9 (CertLedger -> Sepolia Blockchain): Sends smart contract calls executing cryptographic anchoring transactions.
*   Flow 10 (Sepolia Blockchain -> CertLedger): Returns transaction proofs (block numbers, transaction hashes, gas receipts).

3. DATA FLOW DIAGRAM (DFD) LEVEL 1 — SYSTEM PROCESSES IN-DEPTH
================================================================================

The DFD Level 1 breaks open the CertLedger system boundary, representing the 
internal flow of data through specific processes, MongoDB tables (D1), and 
blockchain states (D2).

3.1 Processes & Functions
-------------------------
*   1.0 User Auth & Session Management:
    - Authenticates University Admins, Students, and System Admins.
    - Reads/writes to D1 (MongoDB) to compare bcrypt credentials and issue JSON Web Tokens (JWT).

*   2.0 Certificate Issuance & Anchoring:
    - Receives certificate details from University Admins.
    - Generates a SHA-256 cryptographic hash of the record.
    - Connects to D2 (Sepolia Blockchain) via ethers.js to execute smart contract transactions.
    - Backs up receipt details (block number, tx hash) to D1 (MongoDB).
    - Triggers the next subprocess (3.0) for student access.

*   3.0 Student Account Generation:
    - Automatically triggered when Process 2.0 succeeds.
    - Generates a unique Student User Node in D1 (MongoDB) using the student's ID.
    - Computes a secure temporary password.
    - Links the generated account back to the issued certificate.

*   4.0 Blockchain Verification:
    - Accepts verification requests from Public Employers or Students.
    - Fetches the local record metadata from D1 (MongoDB).
    - Fetches the immutable record state from D2 (Sepolia Blockchain) using the certificate's hash.
    - Performs an active comparison: If the local database hash matches the blockchain state, a "Genuine" result is output. If a mismatch or absence is detected, an anomaly trigger is generated.

*   5.0 Node Account Auditing:
    - Handles System Admin functions.
    - Suspends/activates nodes in D1 (MongoDB).
    - Executes on-chain authorization of university wallets in D2 (Sepolia Blockchain).

*   6.0 Anomaly Tracking:
    - Active security scanner process.
    - Collects data when Process 2.0 or 4.0 detects mismatching hashes, duplicate student IDs, or unauthorized issuing nodes.
    - Saves security records to the Anomaly Store in D1 (MongoDB) and raises immediate visual alerts for the System Admin.

3.2 Data Stores
---------------
*   D1: MongoDB Database: Holds user accounts, certificate metadata backups, temporary credentials, and anomaly audit logs.
*   D2: Sepolia Ledger State: Holds the immutable, cryptographic evidence of certificate hashes anchored through the Solidity Smart Contract.

4. UML SEQUENCE DIAGRAM — CERTIFICATE ISSUANCE WORKFLOW
================================================================================

This diagram shows the sequential execution of actions, messages, and calls 
between system layers during a certificate anchoring process.

4.1 Workflow Execution Steps
----------------------------
1.  Initiation: The University Admin fills out the graduation record form on the React Frontend and clicks "Issue Certificate".
2.  API Call: The React Frontend fires a HTTP `POST /api/certificates/issue` to the Node.js Backend, carrying the certificate payload.
3.  Pre-Validation: The Node.js Backend validates fields and queries the MongoDB Database (DB) to ensure the certificateId does not already exist.
4.  DB Verification: MongoDB returns a `null` result, proving no database collision exists.
5.  Hashing: The Backend takes all relevant fields (student name, ID, qualification, grade, date) and computes a SHA-256 hash.
6.  Blockchain Double-Check: The Backend sends a read-only query to the Sepolia Blockchain calling `checkDuplicateHash(hash)`.
7.  Uniqueness Return: The Sepolia Blockchain contract confirms the hash is unique, returning `false` (no duplicate found).
8.  On-Chain Write: The Backend executes a write transaction `issueCertificate(studentId, hash, institution)`.
9.  Mining & Consensus: The Sepolia nodes process, mine, and record the state change into a block.
10. Transaction Receipt: The Sepolia network returns a rich transaction receipt containing the cryptographic transaction hash and block number.
11. Local Backup: The Backend commits the full certificate metadata along with the blockchain receipt to MongoDB.
12. Account Provisioning: The Backend generates a unique student account and saves it to MongoDB along with a secure temporary password.
13. Sync Success: The Backend sends a `201 Created` HTTP response to the React Frontend containing the transaction receipt and student temporary credentials.
14. Frontend Render: The React Frontend displays a highly visual success modal, giving the University Admin immediate feedback along with the Sepolia transaction link.

5. UML CLASS DIAGRAM — CODEBASE STRUCTURAL MODEL
================================================================================

The Class Diagram illustrates the object-oriented structure of the CertLedger 
backend, highlighting attributes, operation signatures, and relations.

5.1 Model Classes (MongoDB Collections via Mongoose)
----------------------------------------------------
*   User:
    - Attributes: id, username, email, password, role, institution, tempPassword, transactionHash, isActive.
    - Methods:
        * isLocked(): Checks if login attempts have exceeded thresholds.
        * incrementLoginAttempts(): Implements security rate-limiting.
        * resetLoginAttempts(): Resets rate-limiting logs on success.
        * toSafeObject(): Strips out hashed password fields before sending data to clients.

*   Certificate:
    - Attributes: id, certificateId, hash, studentName, studentId, institution, qualification, issueDate, grade, issuer, transactionHash, blockNumber, isActive.
    - Static Methods:
        * findByCertificateId(id): Core retrieval by institutional ID.
        * findByHash(hash): Cryptographic query hook.
        * findRecent(limit): Provides dashboard-level analytics.

*   Anomaly:
    - Attributes: id, certificateId, type, reason, severity, reporter, reporterAddress, institution, resolved.
    - Methods:
        * resolve(by): Toggles audit status and registers the resolving System Admin's ID.

5.2 Controller Classes (API Logic Layers)
-----------------------------------------
*   CertificateController:
    - Methods:
        * issueCertificate(req, res): Manages input checks, blockchain calls, and Mongo writes.
        * verifyCertificate(req, res): Manages on-chain vs. local record consistency checking.
        * getCertificateDetails(req, res): Delivers complete, audited metadata.
        * getIssuanceHistory(req, res): Provides university-specific audits.
        * createStudentAccount(id, name, txHash) [Private]: Internal automated account generation pipeline.
        * generateTemporaryPassword() [Private]: Cryptographic password generator.

*   AuthController:
    - Methods:
        * registerUser(req, res): System Admin creation of university nodes.
        * loginUser(req, res): JWT creation and role verification.
        * changePassword(req, res): Enforces student onboarding reset.
        * getUserProfile(req, res): Self-auditing hook.

*   AdminController:
    - Methods:
        * getUsers(req, res): Administrative dashboard data fetch.
        * toggleUserStatus(req, res): Account suspension switch.
        * deleteUser(req, res): Permanent node deletion.
        * resetPassword(req, res): Admin-assisted override password generation.
        * authorizeIssuer(req, res): Executes smart contract calls authorizing a University Admin's wallet address.

5.3 Class Relationships & Associations
--------------------------------------
*   CertificateController -> Certificate (Manages): Uses the model to create, query, and list academic certificates.
*   CertificateController -> User (Creates Student): Intercepts successful certificate uploads to provision a User instance.
*   AuthController -> User (Authenticates): Queries and updates the User model during auth steps.
*   AdminController -> User (Manages Admin & Student Nodes): Modifies and controls physical fields on User instances.
*   Certificate "1" *-- "0..*" Anomaly (Composition): A Certificate holds a composition relationship with Anomalies. An Anomaly cannot logically exist or be tracked without being linked to an underlying academic certificate identity.
