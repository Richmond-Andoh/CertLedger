1. Document Purpose
This PRD defines the complete requirements for a blockchain-based academic certificate verification system. It covers what problem is being solved, what the system will do, who will use it, how each part works, and how success will be measured. It is the single reference document for all design, development, and evaluation work throughout the five-month project.


2. Problem Statement
2.1 Background
In Ghana, academic certificates are the main document used to prove someone's qualifications. Employers use them when hiring. Universities and professional bodies use them for admission. But the systems currently used to issue and verify these certificates are broken in three clear ways.

2.2 Problem 1 — Fake Certificates Are Easy to Make and Hard to Detect
Anyone in Ghana can get a convincing fake academic certificate cheaply. Using basic design software, a fraudster can copy the appearance of a real certificate from any university. When an employer receives it, they have no fast or reliable way to check if it is genuine.

The result is that unqualified people get jobs they should not have. Honest graduates lose opportunities to fraudsters. In critical fields like medicine and engineering, this creates direct public safety risks.

The core issue: There is no fast, accessible way for an employer to check whether a certificate is real.

2.3 Problem 2 — Verification Is Slow and Unreliable
When an employer wants to confirm a certificate is genuine, the only option today is to contact the university directly by phone, letter, or email. This takes seven to fourteen days on average. Many universities do not respond at all. There is no self-service option. Employers are forced to either trust certificates blindly or delay hiring decisions for weeks.

The core issue: There is no instant, independent verification system available to employers in Ghana.

2.4 Problem 3 — Digital Records Can Be Secretly Changed
Some universities now keep records in digital databases. But these databases are centralised — one office, one system, one point of failure. Anyone with admin access can add a fake record, change a grade, or delete an entry. There is no permanent audit trail. If it happens, it is almost impossible to detect.

The core issue: Centralised digital systems can be manipulated from the inside with no audit trail and no detection mechanism.

2.5 The Identity Binding Question
During the initial presentation, the lecturer raised a valid concern: what if someone obtains a real certificate ID and attaches it to a fake document?

The answer is built into the verification design. The system does not verify the certificate ID alone. It verifies the entire certificate content. The SHA-256 hash is computed from all fields combined — name, student ID, institution, qualification, date, and grade. When an employer verifies a certificate, they submit all these details. If a fraudster uses a borrowed ID but has different personal details, the recomputed hash will not match the blockchain record. Verification returns INVALID. The fraud fails.

2.6 Summary
Academic certificates in Ghana can be easily forged, verified too slowly, and digitally manipulated by insiders. This project builds a system that solves all three problems.


3. Project Goals
The following are the clear, simple goals this project must achieve. Every feature, every requirement, and every design decision in this document exists to serve one of these goals.

Goal	What It Means	Why It Matters
Goal 1 Immutable Records	Certificate hashes are stored on a blockchain. Once written, they cannot be changed or deleted by anyone.	Solves Problem 3. Makes insider tampering impossible after issuance.
Goal 2 Instant Verification	Any employer can verify any certificate in under 5 seconds. No account. No fee. No phone call.	Solves Problem 2. Replaces the slow manual process entirely.
Goal 3 Full-Content Hashing	The hash covers every certificate field — not just the ID. Any change to any field makes verification fail.	Solves Problem 1 and the identity binding concern. Fraud cannot pass verification.
Goal 4 Student Visibility	Every student with a certificate can log in and see what was recorded about them.	Students are the subject of every certificate. They must be able to see their own record.

Goal 6 Simple Account Management	The system admin creates university admin accounts. Student accounts are created automatically when a certificate is issued. No open registration.	Keeps the system secure. Only real graduates with real certificates can have a student account.


4. User Roles
The system has four user roles. Each role has a clear job, a clear set of actions, and a clear access level. The design is deliberately simple — no role can do anything that belongs to another role.

Role	Their Job	What They Can Do	How They Get Access
System Admin	Manages the platform and all accounts	Create and deactivate university admin accounts. Reset passwords. View audit logs. Monitor anomaly dashboard.	Pre-configured account. Not publicly accessible.
University Admin	Issues certificates on behalf of their institution	Log in, issue certificates, view issuance history, see anomaly alerts.	Account created by System Admin. Username and password set by System Admin.
Student / Graduate	Views their own certificate record	Log in and view their own certificate details. Read-only. Cannot edit anything.	Account created automatically when their certificate is issued. Credentials given by university admin.
Employer / Verifier	Verifies whether a certificate is genuine	Enter certificate details on the public verification page. Receive VALID or INVALID result.	No account needed. The verification page is public.

4.1 How Student Accounts Work — Explained Simply
Students cannot register themselves. There is no open registration for the student portal. This is a deliberate security decision — if anyone could sign up as a student, they could claim to be a graduate without a real certificate.

Instead, when a university admin issues a certificate, the system automatically creates a student account at the same time. The student ID becomes the username. The system generates a temporary password. The university admin sees these credentials on the confirmation screen and gives them to the student directly.

The student logs in with those credentials. On first login, they are immediately asked to change their password. After that, they can see their certificate record whenever they want.

If a student forgets their password, they contact their university admin. The university admin can request a password reset through their portal. The system admin can also reset any password from the admin panel.

Key design decision: A student account only exists if a real certificate has been issued for that student. This links authentication directly to the blockchain record.


5. System Architecture
The system has four layers. Every layer has one clear job. The layers communicate with each other through REST APIs.


Figure 1 — System Architecture Overview

6. Account Management Flow
This section answers the question of how accounts are created and managed for each role. The flow below shows the relationship between the System Admin, the University Admin, and the Student — and how accounts move from one level to the next.


Figure 2 — Account Management Flow

6.1 Creating a University Admin Account — Step by Step
    1. System Admin logs into the admin panel.
    2. System Admin clicks 'Add University', enters the university name, and assigns a username and temporary password.
    3. System Admin submits. The university admin account is created.
    4. System Admin gives the credentials to the university admin contact.
    5. University admin logs in. They can now issue certificates.

6.2 How a Student Account Is Created — Step by Step
    1. University admin issues a certificate by filling in the certificate details form.
    2. When the certificate is successfully written to the blockchain, the system automatically creates a student account.
    3. The student ID from the certificate becomes the student's username.
    4. The system generates a temporary password and displays it to the university admin on the confirmation screen.
    5. The university admin gives the username and temporary password to the student.
    6. The student logs in. They are immediately forced to set a new password.
    7. The student can now view their certificate record in the student portal.

6.3 System Admin Capabilities
    • Create university admin accounts — entering university name, username, and temporary password
    • Deactivate any university admin account — a deactivated account cannot log in or issue certificates
    • Reset any student's password — if a student cannot log in, the admin can generate a new temporary password
    • View all registered accounts — university admins and students — in a simple management list
    • View the anomaly dashboard — all flagged certificate events


7. Certificate Issuance Flow
This shows what happens when a university admin issues a certificate. Step 6 — the automatic creation of the student account — is new in this version and addresses the student authentication concern directly.


Figure 3 — Certificate Issuance Flow

7.1 Issuance Steps
    1. University admin logs in using their credentials.
    2. Admin enters all certificate details: student name, student ID, institution, qualification, date, and grade.
    3. The backend computes a SHA-256 hash of all fields combined. Every field is included — not just the ID.
    4. The hash is written to the Solidity smart contract on the blockchain testnet. This record is now permanent.
    5. Certificate metadata is saved to MongoDB. Personal data stays off-chain.
    6. The system automatically creates a student account using the student ID as the username and a generated temporary password.
    7. A confirmation is shown to the admin including the certificate ID, transaction hash, and student credentials.
    8. The anomaly detection module runs automatically in the background.

The issuance is atomic. If the blockchain write in Step 4 fails, nothing is saved — neither the MongoDB record nor the student account. Both must succeed together.


8. Certificate Verification Flow
The verification page is public. Any employer can use it at any time without an account or a fee.


Figure 4 — Certificate Verification Flow

8.1 Verification Steps
    1. Employer opens the public verification page on any browser.
    2. Employer enters all certificate details exactly as they appear on the document: name, student ID, institution, qualification, date, and grade.
    3. The system recomputes the SHA-256 hash from the submitted details.
    4. The recomputed hash is compared against the hash stored on the blockchain.
    5. If the hashes match: VALID — full certificate details are displayed. If they do not match: INVALID — a clear error message is shown.

8.2 Why the Identity Concern Is Answered
The employer enters every field — not just the certificate ID. The system recomputes the hash from all those fields. If a fraudster uses a borrowed certificate ID but their name is different, the hash will not match. INVALID is returned. The fraud fails. This is not a workaround — it is the natural result of full-content hashing built into the core design.


9. Student Portal Flow
The student portal is simple. It has one purpose: let a student see what certificate record was issued for them.

9.1 Student Portal Steps
    1. Student navigates to the student portal and enters their username (student ID) and password.
    2. The system authenticates the student and retrieves their certificate record from MongoDB.
    3. The student sees all their certificate details as recorded by their university.
    4. If anything looks wrong — a misspelled name, an incorrect grade — the student contacts their university to investigate. The student cannot edit their own record.

The student portal is read-only by design. Certificate records are written to the blockchain and cannot be changed after issuance. The student is a viewer of their record, not an editor.


10. Anomaly Detection Flow
The anomaly detection module runs automatically after every certificate issuance. It checks for three specific problems and alerts the System Admin if any are found.


Figure 5 — Anomaly Detection Flow

10.1 Three Checks
Check 1 — Duplicate Hash
After the hash is computed, the module checks whether an identical hash already exists on the blockchain. If it does, this is a duplicate issuance. The module blocks it and raises an alert. This prevents the same certificate being issued more than once.

Check 2 — Rate Spike
The module counts how many certificates one institution has issued in the last hour. If the count exceeds a set threshold — for example ten per hour — it flags the activity as a potential rate spike and alerts the admin. The exact threshold is set during testing using simulated data.

Check 3 — Incomplete Record
The module checks that every required field is present and makes sense. If any field is missing, empty, or logically inconsistent — for example a graduation date that falls before the enrolment date — the record is flagged and the issuance is blocked until the issue is corrected.

All events, clean and flagged, are logged with a timestamp and outcome. Flagged records appear on the System Admin dashboard with a plain-English description of what was detected.

15. Constraints and Limitations
15.1 Constraints
Constraint	Description
Time	5 months, single student. Scope is deliberately kept small.
Budget	No real blockchain costs. Ethereum or Polygon testnet used throughout.
Data	No access to real university records. Simulated data used for all testing.

15.2 What This System Does Not Do
Being honest about limitations makes the project stronger, not weaker.

    • It cannot prevent a corrupt university official from issuing a fraudulent but technically valid certificate. The blockchain stops tampering after issuance — it cannot control what is entered in the first place. This is a known limitation of all blockchain credential systems.
    • It does not biometrically verify that the person presenting a certificate is the person it was issued to. Full-content hashing is the practical solution used here. Biometric extension is future work.
    • It requires internet access. Issuance and verification both need connectivity to reach the blockchain testnet.
    • It is a prototype evaluated on simulated data — not deployed at a real university.
